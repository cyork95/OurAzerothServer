# Task: Implementation Research for 6-Player Party Support

**Context:**
A request has been made to increase the party size from 5 to 6 in our AzerothCore (WotLK 3.3.5a) server. This research prompt summarizes the feasibility and technical constraints discovered by Jules to guide your implementation or proof-of-concept.

**Technical Findings:**

1.  **Server-Side Constraints (AzerothCore):**
    - **Constant:** `MAXGROUPSIZE` is defined as `5` in `src/server/game/Groups/Group.h`.
    - **Raid Impact:** `MAXRAIDSIZE` (40) depends on `MAXGROUPSIZE` for subgroup calculations (`MAX_RAID_SUBGROUPS = MAXRAIDSIZE / MAXGROUPSIZE`). Changing this to 6 would result in 6.66 subgroups, which may cause crashes or logic errors in raid indexing.
    - **Loot & XP:** Distribution logic across the core relies on these constants. Increasing it requires a global audit of `MAXGROUPSIZE` usage.

2.  **Client-Side Constraints (WotLK 3.3.5a Client - HARD LIMITS):**
    - **Unit IDs:** The client binary only recognizes `party1` through `party4`. There is no `party5`. Target/Focus macros and addon `UnitID` checks will fail for a 6th member.
    - **UI Frames:** The default `Interface/FrameXML/PartyMemberFrame.xml` only defines 4 member frames (plus the player). Adding a 5th would require a custom addon to create and anchor a new frame, but it would have no native `UnitID` to bind to.
    - **Opcodes:** `SMSG_PARTY_MEMBER_STATS` and other group-related packets might overflow or be rejected by the client if they contain more data than the hardcoded buffer for 5 members.

**Recommended Path for Antigravity:**

Instead of trying to force a 6-man "Party," consider a "Pseudo-Party" via Raid structures:
- **Automation:** Implement a script/module that automatically converts a 5-man group to a Raid when a 6th member is invited.
- **UI Shim:** Use a client-side addon (like Grid2 or VuhDo) that can display Raid members in a compact "party-like" layout.
- **Scaling:** Leverage `mod-autobalance` to ensure dungeons/raids scale correctly for exactly 6 players.

**Action Items for Antigravity:**
1. Verify if `SMSG_GROUP_LIST` can be "spoofed" to the client to show 6 members without crashing.
2. Investigate if a "hidden raid" mode is possible where the UI behaves like a party but the backend uses raid logic.
3. Assess the impact of changing `MAXGROUPSIZE` to 6 on the `Group` class and its database persistence.
